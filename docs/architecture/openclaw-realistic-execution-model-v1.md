## 1. 核心原则

### 1.1 现实主义
- 只宣传和使用真实可用的能力
- 设计目标明确标注，不作为当前能力
- Fallback 机制必须存在且可用

### 1.2 可验证性
- 能力必须有验证证据
- 没有 ACK 不算派工成功
- 没有 worker 不算 Agent 可执行

### 1.3 铁律
- **没有验证证据不算可用能力**
- **子 Agent 在 worker/ACK 机制实现前，不得作为真实执行体**
- **message_bus 只做状态副本**
- **Codex/GPT/Human-Tool 是当前真实 fallback**

## 4. Fallback 机制

### 4.1 任务分类与 Fallback 策略

| 任务类型 | 首选方案 | Fallback 方案 | 备注 |
|---------|---------|--------------|------|
| **代码执行/本地验证** | Codex | Human-Tool > GPT | Codex 优先，人工验证，GPT 兜底 |
| **架构判断/方案设计** | GPT | Codex > Human-Tool | GPT 擅长架构，Codex 审查，人工兜底 |
| **账号/凭证/系统设置** | Human-Tool | GPT > Codex | 必须人工，GPT 指导，Codex 审查 |
| **记忆沉淀/报告归档** | CEO | Human-Tool > GPT | CEO 主导，人工执行，GPT 辅助 |
| **代码审查** | Codex | GPT > Human-Tool | Codex 优先，GPT 审查，人工兜底 |
| **补丁生成** | Codex | GPT > Human-Tool | Codex 生成，GPT 验证，人工应用 |
| **复杂分析** | GPT | Codex > Human-Tool | GPT 分析，Codex 验证，人工兜底 |

### 4.2 Fallback 流程
```
任务到达
→ CEO 判断类型
→ 选择首选方案
→ 检查首选方案可用性
→ 可用 → 执行
→ 不可用 → Fallback 方案
→ 记录执行结果
→ 写入 memory
```

### 4.3 成本控制
- **Codex**：昂贵，只用于关键任务
- **GPT**：最贵，只用于复杂问题
- **Human-Tool**：免费，用于直接执行
- **优先使用**：Human-Tool > Codex > GPT

## 5. 能力验证机制

### 5.1 验证原则
- 能力必须有验证证据
- 没有验证不算可用
- 定期重新验证

### 5.2 验证方法
- **CEO**：检查决策和记忆写入
- **Codex**：检查审查报告和补丁
- **GPT**：检查方案和代码
- **Human-Tool**：检查执行结果
- **message_bus**：检查状态记录
- **memory**：检查记忆文件
- **qianchen-os-docs**：检查报告文件

### 5.3 验证频率
- **每次使用后**：验证执行结果
- **每日**：检查关键能力状态
- **每周**：全面验证所有能力
