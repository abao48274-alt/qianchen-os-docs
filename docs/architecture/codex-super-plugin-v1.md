## 1. 定位与基线

### 1.1 核心定位
**Codex 是 OpenClaw 的真实工程外挂，而不是子 Agent、不是 CEO、不是 message_bus worker。**

### 1.2 基于现有基线
1. **《OpenClaw 现实执行模型 v1》**：明确真实可用能力和设计目标
2. **《CEO 重要消息交付协议 v1》**：三段式交付：文件 → 通知 → 记忆沉淀
3. **《子 Agent 派工机制验证失败归档报告》**：没有 ACK 不算派工，没有 worker 不算可执行

### 1.3 组织架构
- **CEO（千尘）**：判断、拆解、授权、验收、记忆沉淀、用户汇报
- **Codex（工程外挂）**：代码审查、补丁草稿、本地验证、diff、测试建议
- **message_bus**：只做状态副本
- **qianchen-os-docs**：Codex 报告事实源
- **memory**：CEO 长期记忆索引
- **子 Agent**：在 worker/ACK 机制实现前，不作为真实执行体

## 2. Codex 的角色边界

### 2.1 能做什么
1. ✅ **代码审查**：分析代码质量、安全漏洞、性能问题
2. ✅ **补丁草稿**：生成修复补丁（.patch 文件）
3. ✅ **本地验证**：提供测试建议、验证命令
4. ✅ **diff 生成**：生成代码差异对比
5. ✅ **测试建议**：提供测试用例、测试策略
6. ✅ **PR 审查**：审查 PR 或 diff
7. ✅ **风险分析**：识别潜在风险和问题

### 2.2 不能做什么
1. ❌ **不直接改 main**：不能直接修改主分支
2. ❌ **不自动 git push**：不能自动推送代码
3. ❌ **不碰敏感文件**：不接触 token/.env/openclaw.json
4. ❌ **不碰原始数据**：不接触 V5/memory 原始数据（除非任务明确授权）
5. ❌ **不执行危险操作**：不执行删除、清库、重装、kill 大范围进程
6. ❌ **不自动重试**：命中 usage limit 进入 cooldown，不自动重试
7. ❌ **不替代 CEO 决策**：只提建议，不拍板
8. ❌ **不直接写 message_bus**：只输出文件，由 CEO 或 tools_agent 写状态副本

### 2.3 什么时候该用 Codex
1. **P0/P1 故障**：系统故障、服务中断
2. **跨仓库改动**：涉及多个仓库的代码变更
3. **安全敏感**：涉及安全、token、权限、数据丢失
4. **PR 合并前**：代码合并前的终审
5. **子 Agent 连续失败**：执行失败两次后的复核
6. **复杂代码审查**：逻辑复杂、代码量大
7. **关键函数重写**：核心功能重写
8. **事故复盘**：故障后的复盘分析

### 2.4 什么时候不该用 Codex
1. **普通聊天**：日常对话、问候
2. **简单文档整理**：格式调整、文字修改
3. **普通记忆写入**：日常记忆更新
4. **低风险 UI 文案**：界面文字、提示信息
5. **小 bug 初筛**：简单 bug 的初步筛选
6. **日常状态检查**：系统状态、服务状态
7. **简单格式调整**：代码格式化、空格调整

## 3. Codex 任务类型

### 3.1 read-only（只读审查）
**用途**：分析代码、识别问题、提供审查意见

**输入**：
- 代码文件
- 审查范围
- 审查重点

**输出**：
- 审查报告
- 问题列表
- 风险分析
- 改进建议

**特点**：
- 不生成代码
- 不修改文件
- 只提供分析和建议

### 3.2 patch-only（生成补丁草稿）
**用途**：生成修复补丁、函数替换代码

**输入**：
- 问题描述
- 目标文件
- 修复范围

**输出**：
- 补丁文件（.patch）
- 函数替换代码
- 应用说明

**特点**：
- 只生成补丁
- 不直接应用
- 需要人工或工具应用

### 3.3 local-verify（本地验证/测试建议）
**用途**：提供验证命令、测试建议

**输入**：
- 验证目标
- 测试范围
- 环境信息

**输出**：
- 验证命令
- 测试用例
- 预期结果
- 验证步骤

**特点**：
- 只提供建议
- 不执行验证
- 需要人工或工具执行

### 3.4 PR-review（审查 PR 或 diff）
**用途**：审查代码变更、评估影响

**输入**：
- PR 或 diff 内容
- 变更范围
- 关联问题

**输出**：
- 审查报告
- 变更分析
- 风险评估
- 合并建议

**特点**：
- 专注于变更
- 评估影响
- 提供合并建议

## 4. 标准任务单格式

### 4.1 任务单结构
```yaml
task_id: "CODEX-[YYYYMMDD]-[序号]"
task_type: "read-only|patch-only|local-verify|PR-review"
repo: "仓库路径"
target_files: ["文件路径列表"]
scope: "审查/修复范围描述"
forbidden: ["禁止操作列表"]
acceptance_criteria: ["验收标准列表"]
timeout: "超时时间（秒）"
cost_limit: "成本限制（美元）"
output_path: "输出文件路径"
report_path: "报告文件路径"
```

### 4.2 字段说明

**task_id**
- 格式：`CODEX-[YYYYMMDD]-[序号]`
- 示例：`CODEX-20260514-001`
- 用途：唯一标识任务

**task_type**
- 类型：`read-only`、`patch-only`、`local-verify`、`PR-review`
- 必填：是
- 用途：定义任务类型

**repo**
- 格式：仓库相对路径或绝对路径
- 示例：`/home/administrator/.openclaw/workspace/ai-company`
- 用途：指定工作仓库

**target_files**
- 格式：文件路径数组
- 示例：`["src/app.js", "src/utils.js"]`
- 用途：指定目标文件

**scope**
- 格式：文本描述
- 示例：`"审查 TypeScript 编译错误，提供修复建议"`
- 用途：定义任务范围

**forbidden**
- 格式：禁止操作数组
- 示例：`["git push", "rm -rf", "curl without timeout"]`
- 用途：明确禁止的操作

**acceptance_criteria**
- 格式：验收标准数组
- 示例：`["报告包含问题分析", "补丁能正常应用", "验证命令可执行"]`
- 用途：定义完成标准

**timeout**
- 格式：秒数
- 示例：`300`（5分钟）
- 用途：设置超时时间

**cost_limit**
- 格式：美元金额
- 示例：`0.50`
- 用途：设置成本限制

**output_path**
- 格式：文件路径
- 示例：`qianchen-os-docs/patches/2026-05-14-fix.patch`
- 用途：指定输出文件路径

**report_path**
- 格式：文件路径
- 示例：`qianchen-os-docs/reports/2026-05-14-codex-review.md`
- 用途：指定报告文件路径

## 5. Codex 调用协议

### 5.1 调用流程
```
1. CEO 判断需要 Codex 审查
2. CEO 创建任务单文件
3. CEO 用短消息通知 tools_agent 调用 Codex
4. tools_agent 读取任务单并调用 Codex
5. Codex 执行任务并输出文件
6. CEO 用短消息读取报告
7. CEO 写入 memory/YYYY-MM-DD.md
8. message_bus 只写状态副本
9. Telegram 只发摘要和链接
```

### 5.2 三段式交付（基于《CEO 重要消息交付协议 v1》）

**第一段：写入持久化报告文件**
- 文件路径：`qianchen-os-docs/codex-reports/`
- 文件格式：标准 Markdown
- 内容要求：完整分析、证据、建议、验证结果

**第二段：短消息通知 CEO 读取**
- 消息格式：`Codex 任务完成：[简短描述]，报告：[文件路径]`
- 消息长度：不超过 200 字符
- 必须包含：任务类型 + 文件路径

**第三段：CEO 写入当天 memory**
- 文件路径：`memory/YYYY-MM-DD.md`
- 内容要求：摘要 + 链接，不整篇复制
- 格式：`### [时间] [任务类型] 已读取\n**核心结论**：[结论]\n**报告路径**：[路径]`

### 5.3 状态同步
- **message_bus**：只写状态副本，不作为主交付
- **Telegram**：只发摘要和链接，不发送完整报告
- **状态类型**：`codex_review`、`codex_cooldown`、`idle`

## 6. 安全规则

### 6.1 禁止的操作
1. ❌ **不直接改 main**：不能直接修改主分支
2. ❌ **不自动 git push**：不能自动推送代码
3. ❌ **不碰敏感文件**：不接触 `openclaw.json`、`*.token`、`*.env`
4. ❌ **不碰原始数据**：不接触 `V5/`、`memory/` 原始数据（除非任务明确授权）
5. ❌ **不执行危险操作**：不执行 `rm -rf`、`kill`、`reboot`、`shutdown`
6. ❌ **不执行删除**：不执行大规模删除操作
7. ❌ **不执行清库**：不执行数据库清空操作
8. ❌ **不执行重装**：不执行系统重装操作

### 6.2 命令安全
1. ✅ **所有 curl 必须有 timeout**：`curl --max-time 30`
2. ✅ **所有命令必须有超时**：`timeout 30s command`
3. ✅ **禁止无限循环**：所有循环必须有退出条件
4. ✅ **禁止后台进程**：不创建长期运行的后台进程

### 6.3 文件安全
1. ✅ **只读模式**：默认只读，不修改文件
2. ✅ **补丁模式**：只生成补丁，不直接应用
3. ✅ **输出到指定目录**：所有输出到 `qianchen-os-docs/` 相关目录
4. ✅ **备份原文件**：修改前备份原文件（如适用）

## 7. 验收规则

### 7.1 必须满足（全部通过）
1. ✅ **CEO 必须验收 Codex 输出**：CEO 亲自检查输出质量
2. ✅ **没有报告文件不算完成**：必须有完整的报告文件
3. ✅ **没有验证命令/验证结果不算完成**：必须提供可执行的验证方法
4. ✅ **没有写入 memory 不算 CEO 已吸收**：CEO 必须将摘要写入 memory
5. ✅ **没有明确"未完成事项"不算合格报告**：必须说明遗留问题

### 7.2 验收流程
```
1. Codex 输出报告和文件
2. CEO 读取报告
3. CEO 检查验收标准
4. CEO 写入 memory
5. CEO 标记任务完成
6. message_bus 更新状态
```

### 7.3 验收标准
**报告质量**：
- 包含完整分析
- 包含具体证据
- 包含可执行建议
- 包含验证方法

**文件质量**：
- 路径符合规范
- 格式正确
- 内容完整
- 无敏感信息

**安全合规**：
- 无禁止操作
- 命令有超时
- 无敏感文件访问
- 无危险操作

## 8. Codex cooldown 机制

### 8.1 触发条件
1. **usage limit**：API 调用达到限制
2. **连续失败**：任务连续失败 2 次
3. **超时**：任务执行超时
4. **越界尝试**：尝试执行禁止操作

### 8.2 cooldown 后行为
1. **停止重试**：不自动重试任务
2. **转 Fallback**：转 GPT/Human-Tool fallback
3. **记录状态**：记录到 memory 和 qianchen-os-docs
4. **通知 CEO**：通过短消息通知 CEO

### 8.3 cooldown 恢复
1. **自动恢复**：cooldown 时间结束后自动恢复
2. **手动恢复**：CEO 可手动提前恢复
3. **状态检查**：恢复前检查是否满足恢复条件

### 8.4 记录要求
**memory 记录**：
```
### [时间] Codex 进入 cooldown
**原因**：[触发原因]
**预计恢复**：[恢复时间]
**Fallback 方案**：[GPT/Human-Tool]
```

**qianchen-os-docs 记录**：
- 文件路径：`qianchen-os-docs/reports/YYYY-MM-DD-codex-cooldown.md`
- 内容：触发原因、影响范围、fallback 方案、恢复计划

## 9. 与现实执行模型的关系

### 9.1 在《OpenClaw 现实执行模型 v1》中的位置
- **真实可用能力**：Codex 是当前真实可用 fallback
- **Fallback 策略**：按任务类型路由，Codex 是代码执行/本地验证的首选
- **成本控制**：避免无边界使用，只用于关键任务

### 9.2 与其他组件的关系
- **CEO**：Codex 向 CEO 汇报，由 CEO 验收和决策
- **子 Agent**：子 Agent 不可用时，工程任务优先 Codex/GPT/Human-Tool
- **message_bus**：只做状态副本，不作为主执行通道
- **memory**：CEO 通过 memory 沉淀 Codex 审查结果

### 9.3 执行优先级
1. **代码执行/本地验证**：Codex > Human-Tool > GPT
2. **架构判断/方案设计**：GPT > Codex > Human-Tool
3. **账号/凭证/系统设置**：Human-Tool > GPT > Codex
4. **记忆沉淀/报告归档**：CEO > Human-Tool > GPT
