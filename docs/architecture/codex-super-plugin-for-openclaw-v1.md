## 1. 定位与目标

### 1.1 核心定位
**Codex 是 OpenClaw 的政委 / 总监察官 / 特级工程顾问，不成为主脑，也不作为普通执行员工。**

### 1.2 目标
1. **增强能力**：为 OpenClaw 提供强大的工程审查、风险反对意见、补丁生成、测试建议能力
2. **保持边界**：Codex 不参与决策、不直接修改代码、不自动执行
3. **安全可靠**：所有产出必须经过验证和审批
4. **异步协作**：CEO 不阻塞，任务异步执行

### 1.3 不做什么
- ❌ 不成为主脑（不参与决策）
- ❌ 不直接修改 main 分支
- ❌ 不自动推送代码
- ❌ 不接触 token/.env/openclaw.json
- ❌ 不自动重试（usage limit 进入 cooldown）

└───────────────────────┬─────────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────────┐

└───────────────────────┬─────────────────────────────┘

4. **memory_agent → memory/**：重要结果写入记忆
5. **qa_agent → 验收报告**：检查是否越界

## 3. 分工职责

### 3.1 OpenClaw CEO（千尘）
**职责**：判断、拆解、派工、验收、记忆沉淀

**具体工作**：
1. **判断**：分析需求，决定是否调用 Codex
2. **拆解**：将大任务拆解为小任务
3. **派工**：分配给 tools_agent 执行
4. **验收**：检查产出质量和边界
5. **记忆沉淀**：读取产出文件，写入 memory

**禁止**：
- 亲自调用 Codex API
- 直接修改 Codex 产出
- 阻塞等待 Codex 完成

### 3.2 Codex（政委 / 总监察官 / 特级工程顾问）
**职责**：方案审查、风险反对意见、代码复核、补丁草稿、测试建议

**能力范围**：
1. **代码审查**：分析代码质量、安全漏洞、性能问题
2. **补丁草稿**：生成修复补丁（.patch 文件）
3. **测试建议**：提供测试用例、测试策略
4. **修复建议**：在 CEO 批准后生成最小补丁或修复建议

**限制**：
- 只读模式：默认不直接修改文件
- 补丁模式：只生成 .patch 文件或隔离 diff
- 异步执行：不阻塞 CEO
- 安全边界：不接触敏感文件
- 决策边界：不拍板、不替 CEO 承诺、不替 CEO 写长期 memory

### 3.3 tools_agent（生钱官）
**职责**：调用 Codex、记录日志、处理 cooldown

**具体工作**：
1. **调用 Codex**：使用 codex_runner skill
2. **记录日志**：写入 `ai-company/codex-logs/`
3. **处理 cooldown**：遇到 usage limit 进入冷却
4. **状态同步**：更新 message_bus 状态副本

**工具**：
- `codex_runner` skill
- `ai-company/codex_exec.sh`
- `ai-company/codex_readonly.py`

### 3.4 memory_agent（伴侣内核）
**职责**：把重要结果写入 memory

**具体工作**：
1. **读取产出文件**：从 `reports/` 目录读取
2. **提取关键信息**：摘要、结论、建议
3. **写入 memory**：更新 `memory/YYYY-MM-DD.md`
4. **建立链接**：确保能通过 memory 找到原始产出

### 3.5 qa_agent（测试运维官）
**职责**：验收 Codex 输出是否越界

**具体工作**：
1. **边界检查**：是否接触敏感文件
2. **格式检查**：产出格式是否符合规范
3. **质量检查**：产出质量是否达标
4. **安全检查**：是否有安全风险

## 4. 工作流程

### 4.1 标准流程
```
1. CEO 判断 → 高风险或关键节点需要 Codex 协助
2. CEO 拆解 → 创建任务描述文件
3. CEO 派工 → tools_agent 短消息通知
4. tools_agent 调用 Codex → 只读 + 补丁模式
5. Codex 产出 → 写入 reports/patches/
6. memory_agent 写入 memory/ → 摘要 + 链接
7. qa_agent 验收 → 检查边界和质量
8. CEO 验收 → 读取产出，做出决策并派工
```

### 4.2 异步流程
```
1. CEO 派工 → 不等待完成
2. tools_agent 异步调用 → 后台执行
3. 完成后通知 → 短消息通知 CEO
4. CEO 随时读取 → 不阻塞其他工作
```

### 4.3 Cooldown 处理
```
1. 遇到 usage limit → 进入 cooldown
2. 记录 cooldown 状态 → message_bus
3. 不自动重试 → 等待人工干预
4. CEO 决策 → 是否继续或放弃
```

## 5. 安全规则

### 5.1 禁止接触的文件
- `openclaw.json`（核心配置）
- `*.token` / `*.env`（敏感凭据）
- `~/.ssh/*`（SSH 密钥）
- `~/.gnupg/*`（GPG 密钥）
- `*.key` / `*.pem`（证书文件）

### 5.2 禁止的操作
- 直接修改 main 分支
- 自动 git push
- 直接写入 message_bus（只能通过 tools_agent）
- 自动重试（遇到 error/cooldown 停止）

### 5.3 强制要求
- 产出必须写文件（不直接输出）
- 必须记录日志
- 必须经过 qa_agent 验收
- 必须经过 CEO 读取确认

## 6. 技术规格

### 6.1 文件路径规范
```
qianchen-os-docs/
├── reports/                    # Codex 审查报告
│   └── YYYY-MM-DD-[类型]-[来源].md
├── patches/                    # 补丁文件
│   └── YYYY-MM-DD-[问题]-[文件].patch
└── suggestions/                # 测试建议
    └── YYYY-MM-DD-[功能]-[策略].md

ai-company/
├── codex-logs/                 # 调用日志
│   └── codex-[timestamp].json
└── tasks/                      # 任务描述
    └── T[ID]-[名称].md
```

### 6.2 消息格式规范
**CEO → tools_agent**：
```
Codex 审查任务：[简短描述]
报告文件：[文件路径]
模式：readonly/patch
截止：[时间]
```

**tools_agent → CEO**：
```
Codex 任务完成：[简短描述]
产出文件：[文件路径]
状态：success/cooldown/error
```

### 6.3 日志格式规范
```json
{
  "timestamp": "ISO-8601",
  "task_id": "T[ID]",
  "mode": "readonly|patch",
  "duration_ms": 1234,
  "status": "success|cooldown|error",
  "output_file": "path/to/file",
  "usage": {
    "tokens": 1234,
    "cost": 0.12
  }
}
```

## 9. 风险与缓解

### 9.1 技术风险
- **API 限制**：cooldown 处理机制
- **编码问题**：使用 UTF-8 编码
- **文件权限**：只写入指定目录

### 9.2 安全风险
- **越界访问**：qa_agent 验收检查
- **敏感信息泄露**：禁止接触敏感文件
- **自动执行**：只读 + 补丁模式

### 9.3 流程风险
- **CEO 阻塞**：异步执行机制
- **信息丢失**：三段式交付协议
- **状态不同步**：message_bus 状态副本
