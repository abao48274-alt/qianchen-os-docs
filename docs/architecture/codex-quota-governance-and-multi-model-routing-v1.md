## 1. 问题背景

### 1.1 核心问题
- Codex 额度有限，关键时刻被限流
- 日常任务消耗 Codex 额度，导致紧急故障时无额度可用
- 缺乏分级调用机制，所有任务都调用 Codex
- 缺乏降级机制，usage limit 后无法继续工作

### 1.2 目标
- **治理**：将 Codex 定位为稀缺高级审查资源
- **路由**：建立多模型分级调用机制
- **降级**：usage limit 后自动降级为 DeepSeek
- **预留**：为紧急故障预留 Codex 额度

## 2. 调用等级定义

### 2.1 L0 - 紧急故障（必须 Codex）
**特征**：
- 系统完全不可用
- 数据丢失风险
- 安全漏洞
- P0 级故障

**调用规则**：
- ✅ **必须调用 Codex**
- ✅ 使用预留额度
- ✅ 最高优先级
- ✅ 可以突破 cooldown

**示例**：
- Gateway 崩溃
- Hub 服务中断
- 数据库损坏
- Token 泄露

### 2.2 L1 - 关键审查（优先 Codex）
**特征**：
- P1 级任务
- 跨仓库改动
- PR 合并前终审
- 子 Agent 连续失败两次
- 安全相关变更

**调用规则**：
- ✅ **优先调用 Codex**
- ⚠️ 额度不足时降级为 DeepSeek
- ⚠️ Cooldown 期间降级为 DeepSeek

**示例**：
- 架构重大变更
- 核心功能重写
- 权限系统修改
- 生产环境部署

### 2.3 L2 - 日常工程（默认 DeepSeek）
**特征**：
- P2/P3 级任务
- 单仓库改动
- 代码审查
- 补丁生成
- 测试建议

**调用规则**：
- ✅ **默认调用 DeepSeek**
- ❌ 不调用 Codex
- ✅ DeepSeek 不足时可以跳过

**示例**：
- 日常 bug 修复
- 功能优化
- 代码重构
- 文档更新

### 2.4 L3 - 简单任务（无需 AI）
**特征**：
- 格式调整
- 文字修改
- 状态检查
- 日志查看

**调用规则**：
- ❌ **不调用任何 AI**
- ✅ 子 Agent 直接执行
- ✅ 无需审查

**示例**：
- 代码格式化
- 错别字修正
- 状态查询
- 文件整理

## 4. 额度治理策略

### 4.1 每日额度分配
```
总额度 = Codex 每日可用额度
├── 预留额度 (30%)：紧急故障专用
├── 普通额度 (60%)：L1 任务使用
└── 缓冲额度 (10%)：意外情况
```

### 4.2 预留额度规则
- **专用**：只用于 L0 紧急故障
- **不可挪用**：日常任务不能使用
- **自动补充**：每日重置
- **紧急突破**：CEO 可批准突破预留

### 4.3 普通额度规则
- **L1 任务使用**：关键审查、PR 终审等
- **先到先得**：按申请顺序使用
- **用完即止**：用完后降级为 DeepSeek
- **次日重置**：每日自动重置

### 4.4 缓冲额度规则
- **意外情况**：额度计算误差
- **紧急追加**：CEO 可批准使用
- **监控告警**：使用时通知 CEO

## 5. Usage Limit 处理

### 5.1 Cooldown 机制
```
Codex usage limit 触发
→ 进入 codex_cooldown 状态
→ 记录 cooldown 开始时间
→ 通知 CEO 和相关 Agent
→ 降级为 DeepSeek patch-only
```

### 5.2 Cooldown 状态
- **状态**：codex_cooldown
- **持续时间**：直到额度重置（通常 24 小时）
- **降级策略**：DeepSeek patch-only
- **紧急突破**：CEO 可批准使用预留额度

### 5.3 Cooldown 期间工作
- **L0 任务**：使用预留额度突破 cooldown
- **L1 任务**：降级为 DeepSeek
- **L2 任务**：继续使用 DeepSeek
- **L3 任务**：不受影响

### 5.4 Cooldown 恢复
```
额度重置时间到达
→ 自动退出 codex_cooldown
→ 恢复正常调用
→ 通知 CEO 和相关 Agent
```

## 6. 多模型分工

### 6.1 模型定位
| 模型 | 角色 | 调用等级 | 成本 | 用途 |
|------|------|---------|------|------|
| **Codex** | 政委/总监察官 | L0/L1 | 高 | 关键审查、紧急故障 |
| **DeepSeek** | 副政委/日常参谋 | L1/L2 | 低 | 日常审查、补丁草稿 |
| **GPT** | 外部总顾问 | 特殊 | 最高 | 特别重大决策 |
| **子Agent** | 执行部门 | L2/L3 | 无 | 具体执行 |

### 6.2 调用优先级
```
L0 紧急故障 → Codex (预留额度)
L1 关键审查 → Codex (普通额度) → DeepSeek (降级)
L2 日常工程 → DeepSeek → 跳过
L3 简单任务 → 子Agent 直接执行
```

### 6.3 降级链
```
Codex → DeepSeek → 跳过 → 子Agent 直接执行
```

## 7. CEO 决策流程

### 7.1 调用审批
```
子Agent 提出审查需求
→ CEO 判断调用等级 (L0/L1/L2/L3)
→ CEO 检查额度状态
→ CEO 决定调用模型
→ CEO 派工给 tools_agent
```

### 7.2 紧急突破审批
```
Codex 进入 cooldown
→ L0 紧急故障发生
→ CEO 评估紧急程度
→ CEO 决定是否突破 cooldown
→ CEO 批准使用预留额度
→ tools_agent 执行调用
```

### 7.3 额度监控
- **每日检查**：CEO 检查额度使用情况
- **告警通知**：额度不足时通知 CEO
- **调整权限**：CEO 可调整额度分配

## 8. QA 验收流程

### 8.1 Codex 输出验收
```
Codex 生成报告/补丁
→ qa_agent 检查输出质量
→ qa_agent 验证是否越界
→ qa_agent 检查格式规范
→ qa_agent 验收通过
→ CEO 最终确认
```

### 8.2 验收标准
- **内容质量**：审查意见是否专业
- **格式规范**：是否符合报告格式
- **安全边界**：是否接触敏感文件
- **成本合规**：是否在额度范围内

### 8.3 验收报告
- **文件路径**：`qianchen-os-docs/reports/YYYY-MM-DD-codex-acceptance-qa.md`
- **内容**：验收过程、结果、问题、建议
- **提交**：qa_agent 提交给 CEO

- **更新频率**：实时更新

## 10. 成本控制机制

### 10.1 调用成本监控
- **记录每次调用**：token 使用量、成本
- **每日统计**：总成本、各模型成本
- **告警机制**：成本超预算时告警

### 10.2 成本优化策略
- **默认 DeepSeek**：日常任务使用低成本模型
- **Codex 专用**：只用于关键任务
- **GPT 特殊**：只用于特别重大决策

### 10.3 预算管理
- **每日预算**：设定每日最大成本
- **月度预算**：设定每月最大成本
- **超预算处理**：超预算后降级为更便宜模型
