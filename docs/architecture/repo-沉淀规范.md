# 公共仓库沉淀规范

> 版本：v1.0  
> 日期：2026-05-13  
> 状态：已生效

## 核心原则

**长报告进仓库，聊天里只给摘要和链接。**

## 仓库结构

```
qianchen-os-docs/
├── docs/
│   ├── architecture/          # 架构设计
│   ├── reports/               # 技术报告
│   ├── decisions/             # 决策记录 (ADR)
│   └── roadmap/               # 路线图
├── task-logs/
│   └── public/                # 公开任务日志
├── codex-reports/             # Codex 执行报告
├── README.md                  # 仓库说明
└── .gitignore                 # 忽略规则
```

## 内容边界

### 可上传内容
- 架构设计文档
- 技术报告
- 决策记录 (ADR)
- 任务日志（脱敏）
- Codex 执行报告
- 路线图
- 公开配置示例

### 禁止内容
- API Token / Key
- 密码 / Secret
- 用户隐私聊天原文
- 私有配置文件
- 真实个人敏感数据
- 完整 openclaw.json
- Telegram Bot Token
- 用户 ID（用占位符）

## 敏感信息处理

### API 相关
- 端点 URL：用 `API_ENDPOINT_*` 占位符
- Token：用 `REDACTED` 或删除
- Key：完全删除

### 用户相关
- 用户 ID：用 `USER_ID_*` 占位符
- 聊天内容：不上传原文
- 个人信息：完全删除

### 配置相关
- 密码：删除
- 密钥：删除
- 私有配置：不上传

## 汇报规范

### ceo 汇报格式

**标准格式：**
```
【判断】
一句话说明结论。

【原因】
最多三点。

【下一步】
告诉我你建议怎么做。

【资料】
仓库路径：https://github.com/xxx/qianchen-os-docs/blob/main/xxx.md
```

**示例：**
```
【判断】
异步派工模型方向成立，但 Phase 1 需要先限制并发、增加超时、只做只读任务。

【原因】
1. 核心原则（ceo不阻塞）是对的
2. 实现需要改造现有执行层，现在做会打断 Phase 1 验证
3. 设计已经完整，不会丢失

【下一步】
1. Phase 1 继续用同步模式跑，观察实际阻塞情况
2. 等 Phase 2 启动时，再根据实际数据修订为 v1.1

【资料】
完整设计：qianchen-os-docs/docs/architecture/async-dispatch-v1.md
```

### 长内容处理

**规则：**
- 超过 300 字的内容，先写入仓库
- 聊天中只给摘要和链接
- 不贴全文

**流程：**
1. 创建/更新仓库文档
2. 生成摘要（≤5句话）
3. 摘要 + 仓库路径发给用户
4. 用户直接读取仓库

## 版本管理

### 提交规范
- 每次重要更新提交一次
- 提交信息说明变更内容
- 不频繁提交小改动

### 分支策略
- 主分支：main
- 不创建特性分支（当前阶段）
- 直接提交到 main
