## 1. 组织架构

### 1.1 组织架构
```
投资人/雨陌（最高决策者）
    ↓
CEO 千尘 / 司令员（方向、决策、派工、验收、记忆沉淀）
    ├── 子 Agent 执行部门（pm/coding/qa/tools/memory/ux）
    └── Codex / 政委 + 总监察官 + 特级工程顾问（侧向审查、监督、复核、建议）
```

Codex 不是所有任务的必经执行层，而是 CEO 旁侧的高阶监督与审查力量。普通低风险任务由 CEO 直接派工给子 Agent；高风险任务、关键节点和结果可疑时才召唤 Codex。

### 1.2 角色定位
- **CEO 千尘**：司令员 - 定方向、拆任务、拍板、派工、验收、写入记忆
- **Codex**：政委/监察官 - 审查方案、挑错、看风险、检查执行、给补丁、不能拍板
- **子Agent**：各部门执行官 - 具体执行任务

### 1.3 一句话定位
- CEO 负责"决定打哪仗"
- Codex 负责"审查这仗能不能打、打法有没有漏洞、执行有没有走样"
- 子Agent 负责"具体打"

### 1.4 正式运行标准

本组织设计的执行规则以以下标准为准：

- `qianchen-os-docs/docs/standards/codex-commissar-operating-standard-v1.md`

## 2. Codex 职责

### 2.1 可以做
1. ✅ 审查 OpenClaw 提出的方案
2. ✅ 对方案提出反对意见、风险点、遗漏项
3. ✅ 审查任务拆解是否可执行
4. ✅ 审查子 Agent 的执行报告是否可信
5. ✅ 审查代码 diff
6. ✅ 必要时生成修复补丁
7. ✅ 判断是否应该暂停、回滚、重做
8. ✅ 给 CEO 提建议

### 2.2 不可以做
1. ❌ 不直接拍板
2. ❌ 不直接覆盖 CEO
3. ❌ 不直接长期参与普通聊天
4. ❌ 不直接自动改 main
5. ❌ 不直接改 token/.env/openclaw.json
6. ❌ 不无限调用
7. ❌ 不替 CEO 写长期记忆
8. ❌ 不替 CEO 给用户最终承诺

### 2.3 最终决策链
```
Codex 提建议
→ CEO 判断
→ CEO 拍板
→ CEO 派工
→ 子 Agent 执行
→ Codex 必要时复核
→ CEO 汇报投资人
```

## 3. 何时必须调用 Codex

### 3.1 强制召唤条件
1. **P0/P1 故障**：系统故障、服务中断
2. **跨仓库改动**：涉及多个仓库的代码变更
3. **安全敏感**：涉及安全、token、权限、数据丢失
4. **判断不一致**：CEO 和子 Agent 连续两轮判断不一致
5. **执行失败**：子 Agent 执行失败两次
6. **PR 合并前**：代码合并前的终审
7. **方案派工前**：方案要进入自动派工前

### 3.2 可选召唤条件
1. **复杂代码审查**：逻辑复杂、代码量大
2. **关键函数重写**：核心功能重写
3. **事故复盘**：故障后的复盘分析
4. **架构方案审查**：重大架构决策
5. **验收标准制定**：重要功能的验收标准

---

## 4. 何时禁止调用 Codex

### 4.1 不该叫 Codex
1. **普通聊天**：日常对话、问候
2. **简单文档整理**：格式调整、文字修改
3. **普通记忆写入**：日常记忆更新
4. **低风险 UI 文案**：界面文字、提示信息
5. **小 bug 初筛**：简单 bug 的初步筛选
6. **日常状态检查**：系统状态、服务状态

## 5. 会议流程中 Codex 的位置

### 5.1 3D办公室会议安排
```
CEO 千尘：主持会议
pm_agent：提出需求和优先级
coding_agent：提出实现路径
qa_agent：提出验收和风险
tools_agent：提出执行链路

memory_agent：提出历史经验
Codex 政委：最后审查方案，指出漏洞和风险
CEO 千尘：综合意见后拍板
```

### 5.2 会议流程
1. **CEO 开场**：说明目标
2. **子 Agent 各自发言**：各部门汇报
3. **子 Agent 互相质询**：交叉验证
4. **形成候选方案**：汇总意见
5. **Codex 政委审查候选方案**：最终审查
6. **CEO 最终决策**：拍板决定
7. **CEO 创建任务并派工**：分配执行
8. **子 Agent 执行**：具体实施
9. **Codex 抽查执行结果**：监督复核
10. **CEO 汇报投资人**：结果汇报

## 6. 执行链路中 Codex 的位置

### 6.1 标准执行链路
```
CEO 判断任务风险等级
→ 低风险：CEO 直接派工给子 Agent
→ 高风险：CEO 派工给 tools_agent 调用 Codex 审查
→ Codex 生成报告文件
→ CEO 读取报告并决策
→ CEO 派工给子 Agent 执行
→ qa_agent 初验
→ Codex 必要时抽查或终审
→ memory_agent 写入 memory 摘要
→ CEO 最终验收并汇报投资人
```

### 6.2 异步执行原则
- CEO 不阻塞等待 Codex
- Codex 任务异步执行
- 完成后通知 CEO
- CEO 随时读取

### 7.1 三级顾问体系
- **DeepSeek**：副政委/低成本参谋 - 日常审查、初筛、便宜顾问
- **Codex**：政委/总监察官 - 关键终审、高风险修复

### 7.2 调用策略
- **日常**：DeepSeek 审查
- **关键**：Codex 终审
- **重大**：GPT 外部顾问

### 7.3 成本控制
- DeepSeek：便宜，可频繁调用
- Codex：昂贵，只在关键节点
- GPT：最贵，特别重大决策

- **日志**：tools_agent 记录 Codex 调用日志

## 10. 工作模式三档

### 10.1 模式一：只读政委（默认）
- **输入**：只读方案、只读代码、只读日志
- **输出**：风险、漏洞、反对意见、是否建议执行
- **成本**：中等

### 10.2 模式二：修复政委（需批准）
- **输入**：代码、问题描述、修复范围
- **输出**：补丁、diff、验证结果
- **成本**：高
- **条件**：必须 CEO 批准

### 10.3 模式三：DeepSeek 代班（日常）
- **输入**：简单审查任务
- **输出**：初步意见、筛选建议
- **成本**：低
- **场景**：日常审查、初筛
