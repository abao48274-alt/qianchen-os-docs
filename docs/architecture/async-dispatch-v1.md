# OpenClaw 异步派工模型 v1

> **核心原则：ceo 不阻塞，ceo 只派工**
> 
> 千尘是指挥官，不是苦力。
> 指挥官下达命令后继续指挥，不会亲自去挖战壕。

**版本：** v1.0  
**日期：** 2026-05-13  
**作者：** 千尘 🌌  
**状态：** 设计基线，暂不实现

---

## 一、问题诊断

### 现状（错误模型）

```
用户 → ceo → 执行 Codex（阻塞10-60秒） → 返回结果 → 用户
                ↑
            主会话冻结
            用户只能等待
            无法继续沟通
```

### 期望（正确模型）

```
用户 → ceo → 创建任务单 → 派给执行层 → 立刻回复用户
                                    ↓
                              异步执行中...
                                    ↓
                              完成后回报 ceo
                                    ↓
                              ceo 汇报给用户
```

## 二、异步任务队列

### 2.1 任务状态机

```
                    ┌─────────────┐
                    │   queued    │
                    │   等待派工   │
                    └──────┬──────┘
                           │
                           ▼
                    ┌─────────────┐
              ┌─────│   running   │─────┐
              │     │   执行中    │     │
              │     └─────────────┘     │
              │                         │
              ▼                         ▼
     ┌─────────────────┐      ┌─────────────────┐
     │ waiting_review  │      │     failed      │
     │   等待验收       │      │     失败         │
     └────────┬────────┘      └────────┬────────┘
              │                         │
              ▼                         ▼
     ┌─────────────────┐      ┌─────────────────┐
     │      done       │      │   safe_mode     │
     │     完成         │      │   冻结待处理     │
     └─────────────────┘      └─────────────────┘
```

### 2.2 状态定义

| 状态 | 含义 | 可转换到 |
|------|------|----------|
| `queued` | 任务已创建，等待执行层领取 | `running` |
| `running` | 执行层正在处理 | `waiting_review` / `failed` |
| `waiting_review` | 执行完成，等待 ceo 验收 | `done` / `failed` |
| `done` | 任务完成，已汇报用户 | （终态） |
| `failed` | 执行失败 | `queued`（重试）/ `safe_mode` |
| `safe_mode` | 连续失败3次，冻结 | 人工介入 |

## 三、任务单结构

### 3.1 完整任务单

```json
{
  "task_id": "task-20260513-001",
  "title": "检查V5记忆系统健康状态",
  "description": "只读检查 memory/ 目录结构、MEMORY.md 完整性、索引状态",
  
  "created_by": "ceo",
  "assignee": "coding_agent",
  "tool": "codex",
  "mode": "readonly",
  
  "status": "queued",
  "priority": "normal",
  
  "callback": "telegram:USER_ID_1",
  "session_id": "current",
  
  "created_at": "2026-05-13T05:30:00+08:00",
  "started_at": null,
  "finished_at": null,
  
  "retry_count": 0,
  "max_retries": 3,
  
  "summary": null,
  "output": null,
  "error": null,
  "artifact_path": null,
  
  "safety": {
    "sandbox": "read-only",
    "approval": "never",
    "whitelist": ["codex.check_v5"]
  }
}
```

### 3.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | string | 唯一标识，格式 `task-YYYYMMDD-NNN` |
| `title` | string | 人类可读标题 |
| `description` | string | 详细描述，供执行层理解 |
| `created_by` | string | 创建者，固定 `ceo` |
| `assignee` | string | 执行者：`coding_agent` / `tools_agent` / `qa_agent` |
| `tool` | string | 使用工具：`codex` / `scripts` / `manual` |
| `mode` | string | 执行模式：`readonly` / `write` / `review` |
| `status` | string | 当前状态 |
| `priority` | string | 优先级：`low` / `normal` / `high` / `urgent` |
| `callback` | string | 完成后通知渠道 |
| `session_id` | string | 关联会话，用于恢复上下文 |
| `retry_count` | int | 已重试次数 |
| `max_retries` | int | 最大重试次数，默认3 |
| `summary` | string | 执行摘要（完成后填写） |
| `output` | string | 完整输出（可选） |
| `error` | string | 错误信息（失败时填写） |
| `artifact_path` | string | 产物文件路径 |
| `safety` | object | 安全约束 |

## 四、派工协议

### 4.1 ceo 派工流程

```
1. 识别任务类型
   └─ 预计 > 10秒？ → 必须异步
   
2. 创建任务单
   └─ 写入任务队列
   
3. 选择执行层
   ├─ 代码相关 → coding_agent
   ├─ 工具/自动化 → tools_agent
   └─ 验证/测试 → qa_agent
   
4. 立即回复用户（不等待）
   
5. 主会话释放，继续对话
```

### 4.2 派工回复模板

**标准派工：**
```
已派 {执行层} + {工具} 去做{任务描述}，不会改文件。
我这边继续在线，等它回来后给你汇报。
```

**示例：**
```
已派 coding_agent + Codex 去做只读检查，不会改文件。
我这边继续在线，等它回来后给你汇报。
```

## 五、回报协议

### 5.1 执行层回报流程

```
1. 任务执行完成
   └─ 更新任务单：status, summary, output, artifact_path
   
2. 写入任务日志
   └─ 记录到 task-logs/ 目录
   
3. 通知 ceo
   └─ 通过 Hub/事件系统
   
4. 等待 ceo 验收
```

### 5.2 回报模板

**成功回报：**
```
📋 {执行层} 回报：
任务已完成。

发现：
1. ...
2. ...

建议：
...

产物：{artifact_path}
是否进入下一步？
```

**失败回报：**
```
⚠️ {执行层} 回报：
任务执行失败。

错误：{error_message}
重试次数：{retry_count}/{max_retries}

建议：
...
是否重试或调整方案？
```

## 六、安全规则

### 6.1 强制规则

| 编号 | 规则 | 违反后果 |
|------|------|----------|
| R1 | 预计超过 10 秒的任务必须异步 | 任务被拒绝 |
| R2 | ceo 不等待长任务 | 架构违规 |
| R3 | 执行层结果必须写任务日志 | 任务无效 |
| R4 | Telegram/3D办公室显示任务状态 | 状态不透明 |
| R5 | 失败 3 次进入 safe mode | 自动冻结 |
| R6 | 不允许 Codex 直接绕过 ceo 改代码 | 安全红线 |

### 6.2 Safe Mode 触发条件

- 同一任务连续失败 ≥ 3次
- 检测到非白名单调用
- 执行层试图绕过安全约束
- 手动触发（雨陌指令）

## 七、正确分工

### 7.1 ceo / 千尘中枢

| 职责 | 说明 |
|------|------|
| 目标定义 | 理解用户意图，定义任务目标 |
| 派工 | 创建任务单，选择执行层 |
| 授权 | 授予执行层必要权限 |
| 验收 | 检查执行结果，决定是否通过 |
| 复盘 | 总结经验，优化流程 |
| 继续沟通 | 派工后立即回到对话状态 |

### 7.2 coding_agent

| 职责 | 说明 |
|------|------|
| 代码审查 | 分析代码结构、质量、问题 |
| Codex prompt | 构建合适的 Codex 提示词 |
| 代码修复方案 | 提供修复建议、diff 方案 |
| diff 生成 | 生成代码变更建议 |

### 7.3 tools_agent

| 职责 | 说明 |
|------|------|
| Codex Runner | 实际调用 Codex CLI |
| Telegram 回调 | 推送结果到 Telegram |
| 日志 | 记录执行日志 |
| 任务队列 | 管理任务状态 |
| 限流 | 控制并发、成本 |
| safe mode | 触发安全冻结 |

### 7.4 qa_agent

| 职责 | 说明 |
|------|------|
| 验证只读 | 确认无写入操作 |
| 安全 | 检查安全约束 |
| 端到端任务 | 验证完整流程 |
| 结果可信度 | 评估执行结果可靠性 |

### 7.5 Codex

| 职责 | 说明 |
|------|------|
| 代码级审查 | 静态分析、语法检查 |
| 补丁生成 | 生成代码修复补丁 |
| 测试执行 | 运行测试用例 |

**注意：** Codex 是工具臂，不是主脑。不参与决策，只执行代码级操作。

## 十、战略意义

这个异步派工模型不仅是技术改进，更是 **AI 公司操作系统** 的核心机制：

| 传统 AI 助手 | AI 公司操作系统 |
|--------------|----------------|
| 一次只能做一件事 | 多任务并行 |
| 用户必须等待 | 用户继续工作 |
| AI 是工具 | AI 是团队 |
| 同步阻塞 | 异步协作 |
| 无状态 | 有任务队列 |

**这才是真正的"第二大脑"体验。**
